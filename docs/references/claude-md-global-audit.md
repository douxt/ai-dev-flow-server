### P1：总量超标

**问题**：246 行加载量，远超用户级 CLAUDE.md 推荐值 ≤50–100 行 / ~30–40 条指令（60 上限）。Golden CLAUDE.md 模板将 ≥120 行列明为反模式。

**最佳实践对照**：社区共识用户级 ~80-100 行为宜；指令预算 ≤30 条 actionable bullet。当前文件即便只计正文 bullet（~50+），也已显著超标。

**建议**：目标 80-100 行 + ≤30 条 actionable 指令。通过 P2-P10 的内容去重、外置、迁移自然达到。

---

### P2：代码修改安全重复

**问题**：同一组规则（备份 `.bak` / 即时提交 / 禁 `checkout --` / 全局替换先 grep）在 CLAUDE.md 正文中作为完整章节出现（5 项递进子节），同时在 Worktree 禁令表中又以碎片形式出现一次。Hook 注入版独立于文件外但往每次会话注入了第三份——token 层面是三份。

**最佳实践对照**：社区反模式清单第一条即"Duplicate Rules That Waste Budget"。重复规则在已知案例中浪费 35-40% 上下文预算。应合并为单一权威出处。

**建议**：正文保留 3-4 行红线摘要（指令权重高），删除 5 步细节展开和 Worktree 禁令表中的 checkout 碎片。Hook 注入版保留不动（受保护且注入方式不同）。

---

### P3：Worktree 规则散落

**问题**：Worktree 相关内容分布在 4-5 个独立段落，`grep -ni worktree` 命中 30+ 行：

- `### 计划内置 worktree 步骤（硬性）` — 第 82-90 行
- `## Worktree 开发安全准则` — 第 92-141 行（核心原则 + 禁令表 + 基准分支 + 工作流 10 步 + 已知限制）
- `## 多仓库开发（wt）` — 第 157-177 行（wt 命令表 + fallback 对照表）
- `## 文件写入多层防御 > 子代理约束` — 第 148/152/155 行

**最佳实践对照**：同主题应聚合一处的结构原则；长流程（10 步合并铁律、命令参考表）属于按需查阅内容，不应常驻正文。

**建议**：正文合并为单一 Worktree 红线段 ~10 行（隔离开发、禁主分支编辑、禁 `rm -rf`、禁 `cd && git`、基准默认 origin/master）。完整工作流、合并铁律、wt 命令参考、fallback 表外置到 `docs/references/worktree-workflow.md`，正文仅写路径引用（不 @import）。

### P4：长流程内联

**问题**：两项内容完整内联在正文中，但属于"按需查阅"而非"每轮必要"：

- **ADR 格式模板**（第 207-217 行）：8 行 markdown 代码块，包含完整 ADR-NNN 结构。仅在写设计决策时需要。
- **文件写入三层防御细节**（第 144-155 行）：file-guard / bash-firewall / audit-log 三层机制 + chmod 444 + 日志路径 + 子代理约束。实现细节不需要每轮加载，行为红线需要。

**最佳实践对照**："Keep CLAUDE.md focused; link to detailed docs"。行为约束留正文，机制细节外置。

**建议**：ADR 模板移到 `docs/decisions/`（该目录已存在）。文件写入防御正文压为 2-3 行红线（禁绕过 hook、子代理同受约束、违规被 audit-log 记录）。实现细节外置到子文档。

### P6：软语言与装饰

**问题**：5 处需要清理的内容：

- 第 54-55 行：「长会话适配」段——"全程保持规则一致，长对话不弱化约束"是对模型的喊话，无可执行价值
- 第 7 行："排版朴素无特殊格式"——装饰性风格偏好
- 第 11 行："细小细节按最小改动原则处理"——"细小细节"是模糊量词
- 第 57 行："开发方法灵活选择"——标题暗示可选，但其子节含硬约束（Plan Mode 强制审查）
- 第 139 行："建议同时不超过 3 个"——"建议"是 Soft Limit，无法约束模型行为

**最佳实践对照**：反模式——喊话/元规则、Soft Limits vs Hard Rules、模糊量词。每条都应通过"会犯错吗"试金石。

**建议**：删「长会话适配」整段。清理 5 处软语言为确定性表述（或删除"建议不超过"等无效约束）。非硬约束行去粗，仅保留违反即失败的硬约束使用 `**加粗**`。

### P7：CodeGraph 段偏长

**最佳实践对照**：指南类内容要么压缩为触发准则、要么外置。`.claude/rules/` 路径域条件加载不适用于 CodeGraph（全局工具，非按文件类型触发）。

**建议**：正文压缩为 3-5 行（触发准则 + 双调用方式简洁版 + 跳过兜底）。详表 + 初始化话术外置到 `docs/references/codegraph-guide.md`，正文路径引用。不使用 `.claude/rules/` 条件加载（过度设计）。

### P9：关键约束位置靠后

**问题**：当前 CLAUDE.md 章节顺序为：沟通风格 → 代码输出 → 操作边界 → 代码修改安全（含 5 步细节）→ 工具使用 → 踩坑自省 → 长会话适配 → 开发方法 → **Worktree（第 92 行）**。

最核心的安全红线（Worktree 隔离、代码安全）被埋在文档中部，前 90 行是风格、边界、工具等软约束。

**最佳实践对照**：社区共识——关键约束置顶在前 2-3 个 H2。不是模型"尾部衰减"的问题（CLAUDE.md 是系统级拼接，注意力均匀），而是人眼扫读和逻辑权重排序的问题。

**建议**：精简版将 Worktree 红线 + 代码安全红线紧接「沟通风格」之后出现（第 2-3 个 H2），软约束后置。

**最佳实践对照**：去重补漏——计划应覆盖所有段落的归宿决策。

### P11：缺少指令预算计数

**问题**：P1 只设了行数目标（80-100），但行数 ≠ 指令数。一条"操作边界：严格限定修改范围，不擅自扩范围、不私自重构、不额外加功能"——这是 **3 条指令**，不是 1 行。

当前 217 行含 ~50+ 条 actionable bullet，远超社区 30 条上限。

**最佳实践对照**：模型对指令的实际遵守能力约 150-200 条总预算，系统 prompt 占 ~50，CLAUDE.md 获 ~30-40 条安全上限。超标意味着靠后的指令被模型"遗忘"。

**建议**：改良后统计 bullet 数（`grep -cE "^\s*- "`），目标 ≤30 条 actionable bullet。将此作为 P1 的量化锚点，写入诊断报告。
