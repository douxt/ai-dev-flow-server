# AI 辅助编程开发指南 v5.0

> **信息来源**：Matt Pocock Skills v1.1（16.2 万星，750 万次下载）已成为社区事实标准。本手册基于其五命令体系 + 社区最佳实践综合整理。
>
> **v5.0 修订**：全面采用 Matt Pocock v1.1 五命令（/wayfinder, /research, /to-spec, /to-tickets, /implement, /code-review, /grill-with-docs），上下文预算 ≤40%，引导词体系，安全红线。

## 第一部分：核心理念

1. **软件工程基础原则对 AI 同样有效**：小任务、快速反馈、测试驱动、模块化设计。
2. **AI 不是"规格到代码"的编译器**，而是需要对齐和引导的协作伙伴。
3. **人类负责方向与品味，AI 负责实现与测试**。
4. **上下文预算 ≤40%**：每个 ticket 内容不超过上下文窗口 40%（~48K token for 120K window），确保 agent 在智能区内工作。
5. **阶段切换时：清除上下文，信息通过文档传递**。

## 第二部分：核心命令

| 命令 | 用途 | 占比 |
|------|------|:---:|
| `/grill-with-docs` | 基于文档对话澄清需求 | ~95% 默认入口 |
| `/wayfinder` | 多会话大任务决策地图 | ~5%，上下文装不下时 |
| `/research` | 单 Agent 深度调研 | 按需 |
| `/to-spec` | 需求 → 规格（可逆向：代码→spec） | grill 之后 |
| `/to-tickets` | 规格 → 工单拆分 | spec 之后 |
| `/implement` | 工单 → 代码 + 内建 Maker-Checker | ticket 就绪 |
| `/code-review` | 独立子代理审查全部 diff | implement 之后 |

辅助命令：`/tdd`（红-绿-重构）、`/diagnose`（系统化排查）、`/handoff`（任务交接）

## 第三部分：完整工作流程（6 阶段）

### 阶段 1：explore — 需求澄清

收到任务后，**必须先做工作流评估**：

```
📋 建议路径：[标准/轻量/大型/...]
原因：[...]
是否按此执行？
```

| 步骤 | 操作 |
|------|------|
| 1.1 | 评估任务类型，选择路径 |
| 1.2 | 无现有文档 → Plan Mode 出初稿 |
| 1.3 | 有雾 → `/grill-with-docs`（基于初稿/文档对话澄清） |
| 1.4 | 关键结论写入设计笔记，**清除上下文** |

**例外**：简单改动（~30%）跳过此阶段直接 `/implement`。

### 阶段 2：spec — 产出规格

| 步骤 | 操作 |
|------|------|
| 2.1 | 新建会话，读取上阶段产出 |
| 2.2 | 调用 `/to-spec`（附带 Spec 质量宪法） |
| 2.3 | 验证：Ponytail 四问 + 三假设审计 + 5 级验证层级 |
| 2.4 | 产出 `spec.md`，提交仓库，**清除上下文** |

### 阶段 3：tickets — 拆分工单

| 步骤 | 操作 |
|------|------|
| 3.1 | 新建会话，读取 spec.md |
| 3.2 | 调用 `/to-tickets`（附带 Ticket 质量宪法），上下文预算 ≤40% |
| 3.3 | 检查阻塞关系，确认 DAG |
| 3.4 | 产出 `issues/` 目录，提交仓库，**清除上下文** |

### 阶段 4：implement — 实现 + 审查

| 步骤 | 操作 |
|------|------|
| 4.1 | TDD 前置：测试框架可用 → `/tdd` → 最小 ticket 手动验证 |
| 4.2 | `/implement`（逐 ticket：编码 + 内建 Maker-Checker） |
| 4.3 | `/code-review`（独立子代理审查全部 diff） |

**遇到 bug**：调用 `/diagnose`（复现→缩小→假设→验证→修复），禁止直接猜答案。
**重试限制**：2-3 次，超出 → 输出 `[UNRESOLVED]` 转人工。

### 阶段 5：done — 合并 + 复盘

| 步骤 | 操作 |
|------|------|
| 5.1 | 手动验证功能，尤其前端视觉 |
| 5.2 | 审查测试：确认有意义（非假测试） |
| 5.3 | 检查安全风险 |
| 5.4 | 发现问题 → 创建新 ticket（含复现步骤），不直接修改 AI 代码 |
| 5.5 | 复盘：自动提取教训 → 追加 CLAUDE.md # Lessons |

## 第四部分：Git 工作流规范

| 分支 | AI 权限 | 说明 |
|------|---------|------|
| `main` / `master` | **禁止直接推送** | 人类保留唯一写入权限 |
| 特性分支 | AI 可创建和推送 | 命名格式：`ai/<ticket-id>-<short-desc>` |

- **所有 AI 代码变更必须通过 PR 提交**
- PR 必须经过至少一名人类审查后方可合并
- 安全红线标记（auth/payment/crypto/delete/permission）→ **禁止自动合并**

## 第五部分：引导词体系

| 引导词 | 含义 |
|--------|------|
| 先想清楚再做 | 收到任务先评估路径，不直接改代码 |
| 垂直切片 | 每次交付完整的功能切片，不只是代码片段 |
| 假设可证伪 | 所有假设写下后立刻找反例，不等到审查阶段 |
| 证据不声称 | 说"改了 X"必须附带 diff/测试结果 |
| 不扩范围 | 只做 ticket 要求的事，发现相关问题 → 新开 ticket |
| 每个 Bug 都是永久升级 | 修复后必须追加测试 + 更新规则 |


## 第七部分：常见错误流程对比

| 错误做法 | 正确做法 |
|----------|----------|
| 收到任务直接编码 | 先做工作流评估 |
| 阶段切换时使用压缩 | 阶段切换时**清除上下文** |
| 依赖对话记忆传递信息 | **通过文档传递信息** |
| to-tickets 后立即 implement | 先做 TDD 前置准备 |
| AI 先写实现再补测试 | **强制 TDD：先写失败测试** |
| AI 遇到 bug 直接猜答案 | 调用 `/diagnose` 走系统化排查 |
| AI 尝试无限次自我修复 | **限制重试 ≤ 3 次**，超出转人工 |
| AI 直接 push 到 main | **所有变更通过 PR**，人类审查合并 |
| 直接修改 AI 代码 | 创建新 ticket 让 AI 修复 |
| 单 ticket 超过窗口 40% | 拆分成更小粒度的 ticket |
