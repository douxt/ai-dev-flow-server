# Spec 质量宪法 v3.0

> 每次生成、修改 spec 时，agent 必须逐条对照。缺任一项的 spec 不得进入 tickets 阶段。
>
> 基于原 PRD 宪法 11 项 + Ponytail 四问 + 三假设审计 + 5 级验证层级。

**使用方式**：调用 `/to-spec` 时附带本文件作为上下文，agent 逐条自检后在 spec 尾部输出合规表。

## 完整性与结构

| # | 规则 | 核心要求 | 违规后果 |
|---|------|---------|----------|
| 1 | **章节完整** | Problem / Solution / User Stories / Implementation / Testing / Out of Scope 六段必含 | agent 缺少实现指引 |
| 2 | **Risks & Mitigations** | ≥5 项风险，各有缓解措施 | 后期返工补风险分析 |
| 3 | **定量成功标准** | AC 可测量，禁止模糊词；推荐 EARS 格式 | 验收标准不可验证 |
| 4 | **异常路径覆盖** | 空状态 / 错误状态 / 边界条件 / 权限拒绝 4 类 | Agent 自行脑补边界行为 |

## 外部依赖与接口

| # | 规则 | 核心要求 | 违规后果 |
|---|------|---------|----------|
| 5 | **接口签名明确** | 外部依赖写到函数签名/参数/返回值级 | Agent curl 探索 API |

## 内部一致性

| # | 规则 | 核心要求 | 违规后果 |
|---|------|---------|----------|
| 6 | **内外一致性** | 各章节无矛盾；尾部列关联文档同步清单 | 实现时发现矛盾回退 |

## 可执行性

| # | 规则 | 核心要求 | 违规后果 |
|---|------|---------|----------|
| 7 | **工作量估算 ≤5d** | 单 spec ≤5d，超出则拆分 | AFK 上下文超限 |
| 8 | **User Story 可验证** | "As a...I want...so that" 格式，有独立 AC | US 不可测试 |
| 9 | **Out of Scope 不含未来期** | 不含 Phase N+1 内容 | 范围蔓延 |

## 技术决策

| # | 规则 | 核心要求 | 违规后果 |
|---|------|---------|----------|
| 10 | **技术选型有理由** | 每个技术决策有 justification | 随意选型后期推翻 |
| 11 | **架构约束已引用** | 显式列出不可变架构规则 | 实现违反架构设计 |

## Ponytail 四问（写入 spec §Decisions）

每项关键决策必须回答：

| # | 问题 | 核心要求 |
|---|------|---------|
| P1 | **为什么是这个方案？** | 对比至少 1 个被拒绝的替代方案，说明优劣 |
| P2 | **如果错了会怎样？** | 量化失败影响（回滚成本、数据影响、用户体验降级） |
| P3 | **最小可验证单元是什么？** | 最小的代码/测试可验证本决策正确 |
| P4 | **哪些假设如果错了就全错？** | 列出关键前提，标记可验证时机 |

## 三假设审计

生成 spec 后，逐个检查以下三类假设：

| # | 假设类型 | 审计方法 |
|---|---------|---------|
| H1 | **技术假设** | "这个 API 真的返回这个格式吗？" — 查文档/跑 curl 验证 |
| H2 | **行为假设** | "用户真的会这样用吗？" — 找现有数据/日志支持 |
| H3 | **边界假设** | "这个值真的不会超过上限吗？" — 找极端场景反例 |

每项假设写下后立刻找反例，不等到审查阶段。

## 5 级验证层级（写入 spec §Testing）

| 等级 | 类型 | 定义 | 用于 |
|:---:|------|------|------|
| L1 | 确定性 | 已知输入 → 已知输出，exit code / stdout 可断言 | 核心逻辑、算法 |
| L2 | 阈值型 | 目标范围判断：性能 < Xms，覆盖率 > Y% | 性能、覆盖率 |
| L3 | 比较型 | A/B 对比：新版本不比旧版本差 | 重构、优化 |
| L4 | 多准则 | 加权计分：可维护性 40% + 性能 30% + 安全性 30% | 架构决策 |
| L5 | 主观 | 人判断：UX 合理性、代码可读性 | UI、命名、风格 |

Spec 必须标注每条 AC 对应的验证等级。机器可验证的（L1-L3）优先。L4-L5 需要 human-verify。

## 验证方案模板

```markdown
## 验证方案

| AC | 描述 | 等级 | 验证者 | 通过标准 |
|----|------|:---:|:------:|---------|
| AC-01 | ... | L1 | [auto] | pytest test_xxx.py::test_yyy 通过 |
| AC-02 | ... | L2 | [auto] | API 响应 < 200ms（P95） |
| AC-03 | ... | L5 | [human-verify] | 按钮位置符合设计稿 |
| AC-04 | ... | L4 | [decision] | 技术选型评审通过 |
```

## N/A 豁免

- **#8**：纯技术 spec 不含 User Stories 时可标 N/A
- **#5**：无外部依赖的 spec 可标 N/A
- **P1-P4**：无可选方案的简单改动可标 N/A（需说明原因）
