# TDD 就绪门禁

> 触发：/to-tickets 完成后，/tdd 启动前
> 对应 v3.2 tickets 阶段 → tdd 阶段转换
> 目的：确保所有依赖已就绪，测试可以立即开始编写，不等、不卡、不猜测

## 检查项

| # | 检查项 | 说明 |
|:-:|--------|------|
| R1 | **错误码已定义** | 每个受影响的 API 端点有明确的错误响应：HTTP status 码 + body 格式（至少含 code/msg），不是笼统的"返回错误" |
| R2 | **测试数据已指定** | ticket 前置准备中写的是具体 ID/名称/参数值，不是 `<placeholder>` 或"准备一条测试数据" |
| R3 | **API 入口已明确** | E2E/API 探针的请求目标是完整 HTTP URL（含路径和方法），不是文件路径或"调用 XX 接口" |
| R4 | **权限机制已说明** | 若 ticket 涉及权限控制，已说明后端如何读取权限（查哪个字段/调哪个方法），不是"检查权限"四个字 |
| R5 | **测试场景全部可执行** | 测试方案中无"当前环境无法实现"的场景；复杂场景有替代方案（降级为单元测试或 mock） |
| R6 | **测试环境已就绪** | Docker/DB/测试用户/Token 已验证可用（可用命令或手动确认），不是"应该可以用" |

## 通过条件

R1-R6 全部必须通过。任一不通过 → 不允许启动 /tdd，回 spec 或 ticket 补齐。

## 签出确认

```
[/to-tickets 完成后，/tdd 启动前逐条确认]
[ ] R1: grep "错误码\|error response\|status code" spec.md — 每个 API 有明确错误定义
[ ] R2: 每个 ticket 前置准备含具体值（ID/名称/参数），无占位符
[ ] R3: 探针目标为完整 URL（http://host:port/path METHOD）
[ ] R4: 权限相关 ticket 已说明后端检查机制
[ ] R5: 测试场景清单中无可执行性为"❌ 不可行"的项
[ ] R6: Docker/DB/用户/Token 已验证可用
```
