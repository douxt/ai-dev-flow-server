# Gate 2: 产出 PRD（/to-prd）

## ⚠️ 安全规则（不可跳过）

1. 每完成一个步骤，输出结果给用户，**等待用户确认后才能继续下一步**
2. 禁止自动执行多个步骤，禁止跳过用户确认环节
3. 这不是 CC Workflow 脚本——不做后台子代理，全程与用户交互
4. 遇到不确定的判断，询问用户，不自行决策
5. 全程与用户交互，不 spawn 子代理

## 步骤 0：确认当前 Gate 进度

Read `.gate-state`，确认 gate-2 当前状态：
- 如果 `status: passed` → 跳过，告知用户
- 如果 `status: in_progress` 且有中断记录 → 从断点继续
- 确认 gate-1 必须为 `passed`，否则告知用户"请先完成 Gate 1"
- 如果 `status: pending` → 继续步骤 1

## 步骤 1：读 PRD 质量宪法

Read `03-PRD质量宪法.md`（如项目中不存在，Read `ai-dev-flow/03-PRD质量宪法.md`），理解 11 项宪法规则。向用户概述关键规则，确认理解一致。

**等待用户确认后继续。**

## 步骤 2：生成 PRD — 调用 /to-prd

调用 `/to-prd` 技能生成 PRD。要求：
1. 附带 PRD 质量宪法作为上下文
2. PRD 六段齐全：Problem / Solution / User Stories / Implementation / Testing / Out of Scope
3. Risks ≥5 项且有缓解措施
4. AC 定量可测（EARS 格式推荐）
5. 异常路径覆盖 4 类（空状态/错误状态/边界条件/权限拒绝）

**等待 PRD 生成完成，用户审阅 PRD 内容后继续。**

## 步骤 3：出口检查

逐条检查 Gate 2 出口标准（对照 PRD 质量宪法）：

| # | 检查项 | 宪法条目 | 检查方法 |
|:-:|--------|:--------:|------|
| 2.1 | 六段齐全 | #1 | 扫标题确认六段都在 |
| 2.2 | Risks ≥5 + 缓解 | #2 | 数风险条目，每条有缓解 |
| 2.3 | AC 定量可测 | #3 | 读 AC，查"质量好""完整"等模糊词 |
| 2.4 | 异常路径 4 类 | #4 | 逐类确认空状态/错误/边界/权限 |
| 2.5 | 外部依赖接口签名 | #5 | 检查接口写到参数/返回值级（无外部依赖则 N/A） |
| 2.6 | 估算 ≤5d | #7 | 读工时数字 |
| 2.7 | US 有独立 AC | #8 | 检查 US 格式和独立验收条件（纯技术 PRD 则 N/A） |
| 2.8 | Out of Scope 不含 N+1 | #9 | 扫 OOS 内容，确认无 Phase N+1 内容 |
| 2.9 | 架构约束已引用 | #11 | 确认列出了不可变架构规则 |

**全部通过后**：告知用户，更新 `.gate-state` gate-2 status → `passed`。
**不通过**：输出具体缺失项，用户修正 PRD 后重新检查。严重偏离可能需回 Gate 1。
