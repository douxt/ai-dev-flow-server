# Gate 6: AFK 实施

## ⚠️ 安全规则（不可跳过）

1. 每完成一个步骤，输出结果给用户，**等待用户确认后才能继续下一步**
2. 禁止自动执行多个步骤，禁止跳过用户确认环节
3. 这不是 CC Workflow 脚本——不做后台子代理，全程与用户交互
4. 遇到不确定的判断，询问用户，不自行决策
5. 全程与用户交互，不 spawn 子代理

## 步骤 0：确认当前 Gate 进度

Read `.gate-state`，确认 gate-6 当前状态：
- 如果 `status: passed` → 跳过，告知用户
- 如果 `status: in_progress` 且有中断记录 → 从断点继续
- 确认 gate-5 必须为 `passed`，否则告知用户"请先完成 Gate 5"
- 如果 `status: pending` → 继续步骤 1

## 步骤 1：读脚本宪法 + AFK 规范

1. Read `05-脚本质量宪法.md`，回顾 12 项脚本宪法规则
2. Read `06-AFK脚本栈规范.md`，理解 AFK 自动化流程和脚本要求

## 步骤 2：检查 AFK 实施脚本

1. 检查 `ralph-once.sh` 是否存在（项目 `.devflow/scripts/ralph-once.sh`）
2. 如存在且符合 12 项宪法 → 告知用户可直接启动 AFK 循环
3. 如不存在 → 告知用户需根据 issue 列表手动实施（TDD 流程）

**等待用户确认实施策略后继续。**

## 步骤 3：AFK 实施前确认

与用户逐项确认以下就绪条件：

| # | 确认项 | |
|:-:|--------|:--:|
| 6.1 | Issue 已标记 `ready` | |
| 6.2 | 测试文件路径已指定 | |
| 6.3 | mock 数据/E2E 环境已准备 | |
| 6.4 | SDK/依赖版本已锁定 | |
| 6.5 | AFK 重试上限已配置（默认 3 次） | |
| 6.6 | 用户已知 AFK 期间不干预 | |

**全部确认后**：告知用户，更新 `.gate-state` gate-6 status → `passed`。
用户可以离开，CC 进入 AFK 自动消化循环。
**不通过**：补充缺失项，重新确认。
